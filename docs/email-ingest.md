# Email → Reader: `reader@phareim.no`

**Status: Phase 1 built + deployed (2026-07-12).** What exists:
`POST /api/internal/email-ingest` on the Reader (Bearer
`NUXT_EMAIL_INGEST_KEY`), the `reader-email` Worker (`email-worker/` —
deployed, secret set, CI workflow `deploy-email-worker.yml` path-filtered
on `email-worker/**`), pure helpers in `server/utils/emailIngest.ts` +
`email-worker/src/authResults.ts` (both unit-tested). Phase 1 pulled two
Phase-2 items forward because they were cheap: HTML-part rendering and
first-link URL extraction. **Remaining manual step:** enable Email Routing
on the `phareim.no` zone in the Cloudflare dashboard and add the custom
address `reader@phareim.no` → deliver to Worker `reader-email` (the local
API token lacks the DNS/Email-Routing perms to do it headlessly). Still
open for Phase 2: forward-block author recovery, tracking-pixel strip.

Forward any email — a newsletter issue, a paywalled-article-in-your-inbox, a
long announcement — from your registered email address to
**`reader@phareim.no`**, and it appears in your Reader as an unread card.
This captures the one save-for-later channel none of the Found collectors
reach: things that arrive by email and never had a public URL to bookmark.

## The one-line contract

> Mail sent **to** `reader@phareim.no` **from** an address that matches a
> Reader `User.email` becomes an unread article in that user's Found feed.
> Everything else bounces.

This is deliberately a **forward-to-save** flow, not a subscribe-directly
flow. The sender identifies the user, so one shared address serves everyone
with zero configuration — no per-user tokens to mint, no settings page.
(Direct newsletter subscription — giving Substack an address of ours — is a
future extension; see "Later" below. Its `From` is the newsletter, not a
user, so it needs per-user plus-addressing and is out of scope for v1.)

## Architecture

```
newsletter → user's inbox → user hits Forward →
  reader@phareim.no
       │  Cloudflare Email Routing (phareim.no zone, already on Cloudflare)
       ▼
  reader-email Worker            (new, tiny — email() handler)
       │  1. sender ∈ registered users?  else setReject()
       │  2. parse MIME (postal-mime), pick html ?? text part
       │  3. POST /api/internal/email-ingest  (Bearer, HTTPS)
       ▼
  Reader Worker (existing)
       │  resolve user by sender email → resolveFoundFeed()
       │  insertArticleWithContent()   → D1 + R2 + FTS index
       ▼
  card in the Deck / Found tab, searchable, elevatable
```

### Why a separate `reader-email` Worker

Email Routing can only deliver to a Worker with an `email()` handler. The
Reader Worker's entry is generated by Nitro (`cloudflare-module` preset)
and exports only `fetch` — grafting an `email` export onto the generated
entry means patching build output and breaks silently on Nuxt upgrades.
A standalone ~150-line Worker in this repo (`email-worker/`) is deployed
independently, changes never, and talks to the Reader over the same kind
of authenticated HTTP seam the collectors use. The Reader stays the only
writer of its own database (same rule the do-web/write-web siblings follow).

### The Reader-side endpoint

`POST /api/internal/email-ingest` — Bearer `NUXT_EMAIL_INGEST_KEY` (a new
dedicated secret; not `NUXT_CRON_KEY`, so leaking one key never grants the
other's powers).

```jsonc
{
  "senderEmail": "phareim@gmail.com",   // envelope sender, lowercased
  "messageId":   "<abc@mail.example>",  // Message-ID header
  "subject":     "Fwd: Money Stuff: The Index Fund Did It",
  "author":      "Matt Levine",          // best-effort, see Parsing
  "html":        "<div>…</div>",        // preferred body part
  "text":        "…",                    // fallback part
  "receivedAt":  "2026-07-12T07:31:00Z"
}
```

Behavior:

1. Resolve the user: `SELECT id FROM "User" WHERE email = ? COLLATE NOCASE`.
   No match → 403 (the email Worker turns this into `setReject`, so the
   sender gets a bounce instead of silence).
2. `resolveFoundFeed(event, userId)` — same per-user Found feed the
   collectors use.
3. Insert via `insertArticleWithContent` with:
   - `source: 'email'`
   - `guid: email:<message-id>` (Message-ID truncated/sha-256'd past 180
     chars) — forwarding the same mail twice is a no-op, same idempotency
     rule as every other Found source
   - `title`: Subject with `Fwd:`/`Fw:`/`Re:`/`VS:` prefixes stripped
   - `url`: no real URL exists — synthesize `mailto:<message-id-hash>`?
     No: `Article.url` is NOT NULL and the reader's "view original" hides
     when meaningless. Use the **first http(s) link found in the body**
     (usually the newsletter's "view in browser" link — genuinely useful),
     else `https://reader.phareim.no/` as inert filler. `url_norm` of a
     view-in-browser link also gives cross-source dedup against a bookmark
     of the same issue for free.
   - `content`: the rendered body (below)

Landing in the **Found feed** (not a new feed kind) is deliberate for v1:
Found already flows into the Deck, has its own tab, skips the full-text
fetcher (`kind='found'` guard — important, an email body must never be
"upgraded" by scraping some URL), gets FTS-indexed on insert, and supports
all five verbs including elevate-to-SFL. An emailed newsletter *is* "a
thing I found and pushed to my reader." If email volume ever drowns the
social bookmarks, promoting it to a dedicated `kind='mail'` feed ("Letters")
is a one-column change plus a BottomBar decision — not needed now.

## Parsing (in the email Worker)

- **MIME**: [`postal-mime`](https://github.com/postalsys/postal-mime) —
  Workers-compatible, parses `message.raw` into headers + html/text parts +
  attachments.
- **Body pick**: `html` part if present, else `text` run through the same
  paragraphizer convention the Reader already has (send `text` and let the
  Reader-side endpoint call `paragraphize` — it's a pure shared util).
- **Author**: for a forwarded mail, `From` is the user. Best-effort recovery
  of the original sender: scan the top of the body for the standard
  forward block (`From: Matt Levine <…>` / Gmail's
  `---------- Forwarded message ---------`), take the display name. Miss →
  author stays null; the card is still fine.
- **Sanitization**: none server-side, by existing convention — display-time
  DOMPurify in `pages/article/[id].vue` is the security boundary for every
  article body in the system, and email HTML goes through the same gate.
- **Hygiene trims** (cheap, in the Worker): drop `<img>` tags whose
  width/height ≤ 2 (tracking pixels), drop `<style>`/`<script>` blocks
  (DOMPurify would kill them anyway; trimming keeps R2 lean).
- **Attachments**: ignored in v1. (Later: inline `cid:` images could be
  re-embedded as data URIs under a size cap.)

## Security

Layered, outermost first:

1. **Cloudflare Email Routing itself** rejects inbound mail that fails
   both SPF and DKIM before our Worker ever runs, and stamps
   `ARC-Authentication-Results` on what passes.
2. **Sender allowlist = the User table.** The `email()` handler checks
   `message.from` (envelope sender, lowercased) against registered emails
   via the ingest endpoint's 403. Unknown sender →
   `message.setReject('This address only accepts mail from registered Reader accounts')`
   — an honest bounce, not a black hole.
3. **Spoof hardening**: the Worker parses `ARC-Authentication-Results` and
   requires `spf=pass` **or** `dkim=pass` *aligned with the sender domain*
   (gmail.com mail must carry gmail.com auth). Envelope-From spoofing of a
   registered address from an unauthenticated server dies at layer 1
   anyway; this catches the "passed on some unrelated domain" edge.
4. **Bearer seam**: the ingest endpoint accepts only
   `NUXT_EMAIL_INGEST_KEY`; the key lives as a secret in both Workers and
   grants exactly one power (insert an email card for a registered user).
5. **Caps**: raw message > 2 MB → reject ("too large — forward a link
   instead"); > 20 email ingests per user per day (one D1 count against
   `source='email' AND created_at > now-24h`) → 429/reject. Both limits are
   generous for humans and hostile to loops (an auto-forward rule gone
   wrong is the realistic failure, not an attacker).
6. **worst case** someone who controls a registered user's mailbox can
   insert articles into that user's own feed. That is… the feature. No
   cross-user surface exists: sender maps to exactly one account.

## Config & deploy

- **Zone**: `phareim.no` (already on Cloudflare). Dashboard one-time:
  enable Email Routing, add custom address `reader@phareim.no` → deliver to
  Worker `reader-email`. (MX records are added by the enable step —
  verify nothing else on the zone expects to receive mail; today nothing
  does.)
- **New Worker**: `email-worker/` in this repo — `wrangler.toml`
  (`name = "reader-email"`), `src/index.ts`, own `package.json` with
  `postal-mime`. Deployed by the existing GitHub Actions workflow gaining a
  second `wrangler deploy email-worker` step (path-filtered), or manually
  from this host.
- **Secrets**: `EMAIL_INGEST_KEY` on `reader-email`;
  `NUXT_EMAIL_INGEST_KEY` on the Reader Worker (`wrangler secret put`).
  Plus `READER_API_URL = https://reader.phareim.no` as a plain var.
- **Reader repo changes**: one route file
  (`server/api/internal/email-ingest.post.ts`, ~80 lines — it reuses
  `resolveFoundFeed` + `insertArticleWithContent`, which already handle
  R2, `url_norm` dedup, and FTS indexing), one line in `.env.example`,
  CLAUDE.md + `docs/found-feed.md` updates.

## Rollout

1. **Phase 1 — plumbing**: ingest endpoint + email Worker with auth,
   caps, and text-only rendering. Route live, tested by forwarding real
   mail from phareim@gmail.com. Everyone else still bounces politely.
2. **Phase 2 — rendering polish**: HTML part preferred, forward-block
   author recovery, tracking-pixel strip, view-in-browser URL extraction.
3. **Phase 3 (maybe)**: surface `source: 'email'` distinctly — a small
   `— LETTER` mono label on the card, mirroring how Found cards carry
   their origin.

## Later / explicitly out of scope

- **Direct subscription addresses** (`reader+<token>@phareim.no` handed to
  Substack directly): needs per-user token minting + a settings surface on
  `/sources`, and flips the auth model (sender is the newsletter; the
  token identifies the user). The Email Routing + Worker plumbing built
  here is exactly what it would sit on.
- **A dedicated Letters feed/room** (`kind='mail'`): only if email volume
  earns it.
- **Reply-back** ("done", "save to shelf") — cute, not calm.

## Open questions

1. Bounce copy for unknown senders — is a bounce fine, or should unknown
   senders be silently dropped? (Design says bounce: a user who forwarded
   from their non-registered work address deserves to learn why nothing
   appeared.)
2. Should emailed cards auto-save to the shelf instead of landing unread in
   the deck? (Design says deck: forwarding already expressed "read later",
   and the deck *is* the read-later queue; right-swipe is one gesture away.)
3. `reader@` vs `read@` vs `save@` — pick before the dashboard step;
   everything else is name-agnostic.
